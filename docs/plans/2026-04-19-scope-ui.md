# Plan: Scope UI to a single selected event

## Context

Events are currently a top-level sidebar concept: a list view, a detail view with 5 tabs (tasks/availability/print/details/management), and parallel global views (`TasksView`, `MyBookingsView`, `NotificationPreferencesView`) that show everything across events. In practice this deployment pattern is "one community, one server stack", so a user almost never belongs to more than one active event at a time. The current UI makes them pick an event context on every navigation and exposes concepts that don't matter to ordinary attendees.

Goal: make each user have **one selected event** that scopes the whole authenticated experience. Regular users never see an "events list"; admins keep a dedicated management page for full CRUD. When no event is selected (fresh user) or the selected event's `end_date` has passed, redirect the user into a user-facing selection/onboarding flow.

---

## High-level approach

1. Add `selected_event_id` to `User` (nullable FK to `events`). Expose it on `UserProfile` and add a small endpoint to change it.
2. Build **two distinct views** with clearly separate purposes:
    - `SelectEventView` — user-facing selection screen used for onboarding, "change event", and expired-event redirect. Friendly card layout with upcoming-tasks preview per event. Admins see an extra "Create new" button here so a fresh admin landing with zero events can bootstrap one without leaving the flow, but all detailed CRUD (edit/delete/archive/manage managers at the list level) lives on the admin page below.
    - `AdminEventsView` — admin-only management page under admin settings. Replaces the current [EventsView.vue](frontend/src/views/events/EventsView.vue) at a new route; table/list layout with full CRUD, filters, and bulk ops.
3. Remove the top-level `Events` sidebar section. Promote the former detail-view tabs to top-level routes that implicitly scope to the selected event.
4. Scope the existing global views (tasks, bookings, availability, notification prefs) to the selected event — prefer server-side filtering using the `selected_event_id` already stored on the user, so the frontend doesn't need to pass it as a query param everywhere.
5. Add a route-level guard: if user has no selected event, or the selected event's `end_date < today()`, force-redirect to `SelectEventView` with the appropriate mode. Admin-only routes (including `AdminEventsView`) skip the guard so admins can always reach management.
6. Onboarding: a 2-step wizard — step 1 is the selection screen (`SelectEventView`), step 2 is notification channel setup (email/push/telegram toggles + phone number). Existing channel-setup components are reused; no new notification logic.

## Backend changes

### 1. User model + migration

[backend/app/models/user.py](backend/app/models/user.py)

- Add `selected_event_id: uuid.UUID | None` with FK `events.id`, `ondelete="SET NULL"`, indexed.

New Alembic migration under `backend/app/alembic/versions/` that adds the column + FK.

### 2. Schemas

[backend/app/schemas/users.py](backend/app/schemas/users.py)

- Add `selected_event_id: uuid.UUID | None` to `UserProfile`.
- Add new `SelectedEventUpdate` schema (single field: `selected_event_id: uuid.UUID | None`).

### 3. Routes

[backend/app/api/routes/users.py](backend/app/api/routes/users.py)

- New endpoint `PUT /users/me/selected-event` accepting `SelectedEventUpdate`. Validates the event exists and is accessible to the user (published OR user is admin / task_manager / scoped event manager). Persists to `User.selected_event_id`. Returns updated `UserProfile`.
- Extend `_build_user_profile` to include `selected_event_id`.

### 4. Scoped filtering (use user's selected event as the default filter)

Add a small helper (e.g. `app/logic/event_scope.py`) that returns the current user's `selected_event_id`, with an explicit "bypass" path for automated/system code.

Apply it in:

- [backend/app/api/routes/tasks/feed.py](backend/app/api/routes/tasks/feed.py) — default the feed's `event_id` filter to the user's selection when not explicitly provided. Leave query-param override for admin use.
- [backend/app/api/routes/tasks/crud.py](backend/app/api/routes/tasks/crud.py) — same pattern where listing endpoints exist.
- [backend/app/api/routes/bookings.py](backend/app/api/routes/bookings.py) — filter "my bookings" lookups to the selected event by default.
- Notification preferences (wherever per-type preferences are read/written, likely `backend/app/api/routes/notifications*.py`) — per-type preferences become scoped by `(user_id, event_id, notification_type)`. **Global channel toggles (`notify_email`, `notify_push`, `notify_telegram`) stay user-global** — they're the kill-switches the user called out as "except automated notifications".

> ⚠️ If per-type notification preferences are currently keyed only by `(user_id, notification_type)`, scoping them per-event is a schema change in its own right. Verify the existing schema during implementation; if non-trivial, land the core scoping (tasks/bookings) first and treat per-type notification prefs as a follow-up step in the same PR.

### 5. Expiration check

Add an `is_expired` computed property on `Event` (`end_date < today()`) for use by the frontend via the detail endpoint. No hard server-side block — the guard lives in the frontend so admins can still edit expired events.

## Frontend changes

### 1. Regenerate client after backend schema changes

Run `just generate-client` after the backend edits land. Do **not** hand-edit `frontend/src/client/`.

### 2. Auth store

[frontend/src/stores/auth.ts](frontend/src/stores/auth.ts)

- Expose `selectedEventId` (computed from `profile.selected_event_id`).
- Add `setSelectedEvent(id: string | null)` action that calls the new backend endpoint and refreshes `profile`.
- Add `selectedEvent` (ref) that holds the resolved `EventRead` for the selected id, fetched on profile load / change.

### 3. New user-facing view: `SelectEventView`

New file: `frontend/src/views/events/SelectEventView.vue`

One route (`/app/select-event`), behavior driven by a `mode` query param:

- `?mode=onboarding` — 2-step wizard:
    - **Step 1 — Pick event**: card grid of events the user can access. Each card shows name, date range, and a preview of upcoming tasks. Selecting sets `selected_event_id` and advances. Admins see an extra "Create new" card/button that opens a minimal create dialog inline (reuses the current create form from [EventsView.vue](frontend/src/views/events/EventsView.vue)); on success, the new event auto-selects and advances.
    - **Step 2 — Notification channels**: reuse the existing channel cards from [NotificationPreferencesView.vue](frontend/src/views/NotificationPreferencesView.vue) (`PushChannelCard`, `TelegramChannelCard`, `AnimatedMail` email toggle) plus a `phone_number` input. Finish → dashboard.
- `?mode=switch` — just step 1 (no wizard chrome, no step 2), used for manual "change event" from header pill / user menu.
- `?mode=expired` — step 1 with a red/amber banner explaining the previous event ended and the user must pick another.

This view is intentionally **user-focused**: no delete/archive/edit actions, no bulk ops, no filters. It's a friendly picker + onboarding flow. The admin shortcut ("Create new") exists only to avoid the dead-end case where an admin with zero events gets redirected here by the guard.

### 4. New admin view: `AdminEventsView`

New file: `frontend/src/views/admin/AdminEventsView.vue`, mounted at `/app/admin/events`.

This is the full-CRUD replacement for the current [EventsView.vue](frontend/src/views/events/EventsView.vue). Most of its existing logic (filters, create, delete, status badges, date-range filter) moves here verbatim. Differences from the user-facing selection view:

- Table/list layout optimized for management, not card-grid for picking.
- Edit / delete / archive actions inline per row.
- Visible only when `authStore.isAdmin || authStore.isTaskManager`.
- Linked from the admin settings area of the sidebar.
- Selecting a row opens `/app/event-settings?eventId={id}` (i.e., admin can edit any event's details/managers without having to make it their currently selected event).

After both new views ship, delete the old [EventsView.vue](frontend/src/views/events/EventsView.vue).

### 5. Promote former tabs to top-level routes

Current tabs inside [EventDetailView.vue](frontend/src/views/events/EventDetailView.vue) become top-level routes that read the selected event from the auth store:

| Former tab   | New route                          | Notes                                                                                                                                                    |
| ------------ | ---------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Tasks        | `/app/tasks` (already exists)      | Remove the event-list overlay logic; backend already scopes                                                                                              |
| Availability | `/app/availability`                | Wraps existing `EventAvailability` component                                                                                                             |
| Print        | `/app/print`                       | Mobile-only, wraps existing `EventPrint`                                                                                                                 |
| Details      | `/app/event-settings`              | Admin/manager only, wraps existing `EventEditForm`. Accepts optional `?eventId=` (from `AdminEventsView`) to edit any event; defaults to selected event. |
| Management   | `/app/event-settings?tab=managers` | Same view, tabbed — keeps things compact                                                                                                                 |

Delete [EventDetailView.vue](frontend/src/views/events/EventDetailView.vue) once its child components are wired into the new routes.

### 6. Router + guard

[frontend/src/router/index.ts](frontend/src/router/index.ts)

- Add routes above; remove `events` list + `event-detail` routes (and any navigation that links to them).
- In `authGuard`, after `ensureProfile()`, check:
    1. If `selectedEventId` is null and target isn't `select-event` / admin routes → redirect to `/app/select-event?mode=onboarding`.
    2. If `selectedEvent.is_expired` and target isn't `select-event` / admin routes → redirect to `/app/select-event?mode=expired`.
    - Admin routes (`/app/admin/**`, `/app/event-settings`) skip the guard so admins without a selected event can still reach management.

### 7. Navigation cleanup

- [AppSidebar.vue](frontend/src/components/navigation/AppSidebar.vue) lines 128-178: remove the "Events" section entirely. Add entries for the new promoted routes (Availability, Print, Event Settings) and an admin-gated link to `AdminEventsView` under an "Admin" section. Keep Tasks, My Bookings, Inbox.
- [PostAuthHeader.vue](frontend/src/components/layout/postauth/PostAuthHeader.vue): add a compact "Current event" pill/chip showing the selected event name; clicking routes to `/app/select-event?mode=switch`. This is the "always visible" context the user asked for.
- [MobileBottomNav.vue](frontend/src/components/navigation/MobileBottomNav.vue): no change (Tasks/Bookings/Inbox already work).
- [MobileUserMenu.vue](frontend/src/components/navigation/MobileUserMenu.vue): add a "Change event" link routing to `/app/select-event?mode=switch`. Admins additionally see a "Manage events" link routing to `/app/admin/events`.
- [breadcrumb.ts](frontend/src/stores/breadcrumb.ts): drop the "Events > {Name} > {Section}" chain; crumbs are flat now.
- [sidebar.ts](frontend/src/stores/sidebar.ts) + backend `/dashboard/sidebar`: the `events` payload field can be removed; sidebar no longer lists events.

### 8. i18n

- Add keys in both [locales/en/navigation.json](frontend/src/locales/en/navigation.json) and [locales/de/navigation.json](frontend/src/locales/de/navigation.json) for the new nav items, the "change event" link, and the onboarding/expired banners.
- Remove the now-unused `navigation.sidebar.items.events.*` keys (the old top-level events list entry).

### 9. Scoped views cleanup

- [TasksView.vue](frontend/src/views/tasks/TasksView.vue): drop the event overlay fetch (backend now scopes); header shows the selected event name from the store.
- [NotificationPreferencesView.vue](frontend/src/views/NotificationPreferencesView.vue): UI stays the same; per-type prefs are now implicitly scoped by the backend.

## Verification

1. **Backend unit tests**: `just test-backend` — expect new tests for the selected-event endpoint (auth, validation that event is accessible, SET NULL on delete). Existing feed/bookings tests should still pass since the default filter only kicks in when no explicit `event_id` query param is provided.
2. **Manual smoke flow** (`just dev local`, then browser at `http://localhost:5555`):
    - **Fresh user**: log in → redirected to `/app/select-event?mode=onboarding` → pick an event → configure notifications → land on dashboard.
    - **Returning user**: dashboard loads directly; Tasks/Availability/Bookings show only the selected event's data.
    - **Header pill**: shows current event name; clicking routes to `?mode=switch` picker.
    - **Change event**: pick a different event → views refresh with new scope.
    - **Expired event**: set an event's `end_date` to yesterday in the DB → reload → full-screen redirect to `?mode=expired`.
    - **Admin bootstrap (zero events)**: log in as admin with no events in DB → redirected to `?mode=onboarding` → "Create new" button visible → create an event inline → advance through wizard.
    - **Admin management**: navigate to `/app/admin/events` → full table with edit/delete/archive; "Edit" opens `/app/event-settings?eventId=…` without changing the admin's own selected event.
3. **E2E**: extend `pnpm --prefix frontend test:e2e` with an onboarding spec (login → pick event → see tasks scoped). Existing specs that navigated via `/app/events` need updating.
4. **API client regen**: run `just generate-client` after backend changes and check that `frontend/src/client/types.gen.ts` now exposes `selected_event_id` on `UserProfile`.
5. **Lint & typecheck**: `just lint` must pass — expect to fix imports of removed views/routes.
