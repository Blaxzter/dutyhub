# Plan: Scope UI to a single selected event group

## Context

Event groups are currently a top-level sidebar concept: a list view, a detail view with 5 tabs (events/availability/print/details/management), and parallel global views (`EventsView`, `MyBookingsView`, `NotificationPreferencesView`) that show everything across groups. In practice this deployment pattern is "one community, one server stack", so a user almost never belongs to more than one active event group at a time. The current UI makes them pick a group context on every navigation and exposes concepts that don't matter to ordinary attendees.

Goal: make each user have **one selected event group** that scopes the whole authenticated experience. Regular users never see a "groups list"; admins keep a dedicated management page for full CRUD. When no group is selected (fresh user) or the selected group's `end_date` has passed, redirect the user into a user-facing selection/onboarding flow.

> **Terminology note**: a separate follow-up plan will rename `EventGroup → Event`, `Event → Task`, `Slot → Shift` throughout the codebase and UI. This plan keeps the current terminology so diffs stay focused; the rename runs first as its own PR, and this work builds on top of it.

---

## High-level approach

1. Add `selected_event_group_id` to `User` (nullable FK to `event_groups`). Expose it on `UserProfile` and add a small endpoint to change it.
2. Build **two distinct views** with clearly separate purposes:
    - `SelectEventGroupView` — user-facing selection screen used for onboarding, "change group", and expired-group redirect. Friendly card layout with upcoming-events preview per group. Admins see an extra "Create new" button here so a fresh admin landing with zero groups can bootstrap one without leaving the flow, but all detailed CRUD (edit/delete/archive/manage managers at the list level) lives on the admin page below.
    - `AdminEventGroupsView` — admin-only management page under admin settings. Replaces the current [EventGroupsView.vue](frontend/src/views/event-groups/EventGroupsView.vue) at a new route; table/list layout with full CRUD, filters, and bulk ops.
3. Remove the top-level `Event Groups` sidebar section. Promote the former detail-view tabs to top-level routes that implicitly scope to the selected group.
4. Scope the existing global views (events, bookings, availability, notification prefs) to the selected group — prefer server-side filtering using the `selected_event_group_id` already stored on the user, so the frontend doesn't need to pass it as a query param everywhere.
5. Add a route-level guard: if user has no selected group, or the selected group's `end_date < today()`, force-redirect to `SelectEventGroupView` with the appropriate mode. Admin-only routes (including `AdminEventGroupsView`) skip the guard so admins can always reach management.
6. Onboarding: a 2-step wizard — step 1 is the selection screen (`SelectEventGroupView`), step 2 is notification channel setup (email/push/telegram toggles + phone number). Existing channel-setup components are reused; no new notification logic.

## Backend changes

### 1. User model + migration

[backend/app/models/user.py](backend/app/models/user.py)

- Add `selected_event_group_id: uuid.UUID | None` with FK `event_groups.id`, `ondelete="SET NULL"`, indexed.

New Alembic migration under `backend/app/alembic/versions/` that adds the column + FK.

### 2. Schemas

[backend/app/schemas/users.py](backend/app/schemas/users.py)

- Add `selected_event_group_id: uuid.UUID | None` to `UserProfile`.
- Add new `SelectedEventGroupUpdate` schema (single field: `selected_event_group_id: uuid.UUID | None`).

### 3. Routes

[backend/app/api/routes/users.py](backend/app/api/routes/users.py)

- New endpoint `PUT /users/me/selected-event-group` accepting `SelectedEventGroupUpdate`. Validates the group exists and is accessible to the user (published OR user is admin/event_manager/manager of it). Persists to `User.selected_event_group_id`. Returns updated `UserProfile`.
- Extend `_build_user_profile` to include `selected_event_group_id`.

### 4. Scoped filtering (use user's selected group as the default filter)

Add a small helper (e.g. `app/logic/event_group_scope.py`) that returns the current user's `selected_event_group_id`, with an explicit "bypass" path for automated/system code.

Apply it in:

- [backend/app/api/routes/events/feed.py](backend/app/api/routes/events/feed.py) — default the feed's `event_group_id` filter to the user's selection when not explicitly provided. Leave query-param override for admin use.
- [backend/app/api/routes/events/crud.py](backend/app/api/routes/events/crud.py) — same pattern where listing endpoints exist.
- [backend/app/api/routes/bookings.py](backend/app/api/routes/bookings.py) — filter "my bookings" lookups to the selected group by default.
- Notification preferences (wherever per-type preferences are read/written, likely `backend/app/api/routes/notifications*.py`) — per-type preferences become scoped by `(user_id, event_group_id, notification_type)`. **Global channel toggles (`notify_email`, `notify_push`, `notify_telegram`) stay user-global** — they're the kill-switches the user called out as "except automated notifications".

> ⚠️ If per-type notification preferences are currently keyed only by `(user_id, notification_type)`, scoping them per-group is a schema change in its own right. Verify the existing schema during implementation; if non-trivial, land the core scoping (events/bookings) first and treat per-type notification prefs as a follow-up step in the same PR.

### 5. Expiration check

Add an `is_expired` computed property on `EventGroup` (`end_date < today()`) for use by the frontend via the detail endpoint. No hard server-side block — the guard lives in the frontend so admins can still edit expired groups.

## Frontend changes

### 1. Regenerate client after backend schema changes

Run `just generate-client` after the backend edits land. Do **not** hand-edit `frontend/src/client/`.

### 2. Auth store

[frontend/src/stores/auth.ts](frontend/src/stores/auth.ts)

- Expose `selectedEventGroupId` (computed from `profile.selected_event_group_id`).
- Add `setSelectedEventGroup(id: string | null)` action that calls the new backend endpoint and refreshes `profile`.
- Add `selectedEventGroup` (ref) that holds the resolved `EventGroupRead` for the selected id, fetched on profile load / change.

### 3. New user-facing view: `SelectEventGroupView`

New file: `frontend/src/views/event-groups/SelectEventGroupView.vue`

One route (`/app/select-group`), behavior driven by a `mode` query param:

- `?mode=onboarding` — 2-step wizard:
    - **Step 1 — Pick group**: card grid of groups the user can access. Each card shows name, date range, and a preview of upcoming events. Selecting sets `selected_event_group_id` and advances. Admins see an extra "Create new" card/button that opens a minimal create dialog inline (reuses the current create form from [EventGroupsView.vue](frontend/src/views/event-groups/EventGroupsView.vue)); on success, the new group auto-selects and advances.
    - **Step 2 — Notification channels**: reuse the existing channel cards from [NotificationPreferencesView.vue](frontend/src/views/NotificationPreferencesView.vue) (`PushChannelCard`, `TelegramChannelCard`, `AnimatedMail` email toggle) plus a `phone_number` input. Finish → dashboard.
- `?mode=switch` — just step 1 (no wizard chrome, no step 2), used for manual "change group" from header pill / user menu.
- `?mode=expired` — step 1 with a red/amber banner explaining the previous group ended and the user must pick another.

This view is intentionally **user-focused**: no delete/archive/edit actions, no bulk ops, no filters. It's a friendly picker + onboarding flow. The admin shortcut ("Create new") exists only to avoid the dead-end case where an admin with zero groups gets redirected here by the guard.

### 4. New admin view: `AdminEventGroupsView`

New file: `frontend/src/views/admin/AdminEventGroupsView.vue`, mounted at `/app/admin/event-groups`.

This is the full-CRUD replacement for the current [EventGroupsView.vue](frontend/src/views/event-groups/EventGroupsView.vue). Most of its existing logic (filters, create, delete, status badges, date-range filter) moves here verbatim. Differences from the user-facing selection view:

- Table/list layout optimized for management, not card-grid for picking.
- Edit / delete / archive actions inline per row.
- Visible only when `authStore.isAdmin || authStore.isEventManager`.
- Linked from the admin settings area of the sidebar.
- Selecting a row opens `/app/group-settings?groupId={id}` (i.e., admin can edit any group's details/managers without having to make it their currently selected group).

After both new views ship, delete the old [EventGroupsView.vue](frontend/src/views/event-groups/EventGroupsView.vue).

### 5. Promote former tabs to top-level routes

Current tabs inside [EventGroupDetailView.vue](frontend/src/views/event-groups/EventGroupDetailView.vue) become top-level routes that read the selected group from the auth store:

| Former tab   | New route                          | Notes                                                                                                                                                              |
| ------------ | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Events       | `/app/events` (already exists)     | Remove the group-list overlay logic; backend already scopes                                                                                                        |
| Availability | `/app/availability`                | Wraps existing `EventGroupAvailability` component                                                                                                                  |
| Print        | `/app/print`                       | Mobile-only, wraps existing `EventGroupPrint`                                                                                                                      |
| Details      | `/app/group-settings`              | Admin/manager only, wraps existing `EventGroupEditForm`. Accepts optional `?groupId=` (from `AdminEventGroupsView`) to edit any group; defaults to selected group. |
| Management   | `/app/group-settings?tab=managers` | Same view, tabbed — keeps things compact                                                                                                                           |

Delete [EventGroupDetailView.vue](frontend/src/views/event-groups/EventGroupDetailView.vue) once its child components are wired into the new routes.

### 6. Router + guard

[frontend/src/router/index.ts](frontend/src/router/index.ts)

- Add routes above; remove `event-groups` and `event-group-detail` routes (and any navigation that links to them).
- In `authGuard`, after `ensureProfile()`, check:
    1. If `selectedEventGroupId` is null and target isn't `select-group` / admin routes → redirect to `/app/select-group?mode=onboarding`.
    2. If `selectedEventGroup.is_expired` and target isn't `select-group` / admin routes → redirect to `/app/select-group?mode=expired`.
    - Admin routes (`/app/admin/**`, `/app/group-settings`) skip the guard so admins without a selected group can still reach management.

### 7. Navigation cleanup

- [AppSidebar.vue](frontend/src/components/navigation/AppSidebar.vue) lines 128-178: remove the "Event Groups" section entirely. Add entries for the new promoted routes (Availability, Print, Group Settings) and an admin-gated link to `AdminEventGroupsView` under an "Admin" section. Keep Events, My Bookings, Inbox.
- [PostAuthHeader.vue](frontend/src/components/layout/postauth/PostAuthHeader.vue): add a compact "Current group" pill/chip showing the selected group name; clicking routes to `/app/select-group?mode=switch`. This is the "always visible" context the user asked for.
- [MobileBottomNav.vue](frontend/src/components/navigation/MobileBottomNav.vue): no change (Events/Bookings/Inbox already work).
- [MobileUserMenu.vue](frontend/src/components/navigation/MobileUserMenu.vue): add a "Change event group" link routing to `/app/select-group?mode=switch`. Admins additionally see a "Manage event groups" link routing to `/app/admin/event-groups`.
- [breadcrumb.ts](frontend/src/stores/breadcrumb.ts): drop the "Event Groups > {Name} > {Section}" chain; crumbs are flat now.
- [sidebar.ts](frontend/src/stores/sidebar.ts) + backend `/dashboard/sidebar`: the `event_groups` payload field can be removed; sidebar no longer lists groups.

### 7. i18n

- Add keys in both [locales/en/navigation.json](frontend/src/locales/en/navigation.json) and [locales/de/navigation.json](frontend/src/locales/de/navigation.json) for the new nav items, the "change group" link, and the onboarding/expired banners.
- Remove the now-unused `navigation.sidebar.items.eventGroups.*` keys.

### 8. Scoped views cleanup

- [EventsView.vue](frontend/src/views/events/EventsView.vue): drop the event-group overlay fetch (backend now scopes); header shows the selected group name from the store.

1. **Backend unit tests**: `just test-backend` — expect new tests for the selected-group endpoint (auth, validation that group is accessible, SET NULL on delete). 2. **Manual smoke flow** (`just dev local`, then browser at `http://localhost:5173`):
    - **Fresh user**: log in → redirected to `/app/select-group?mode=onboarding` → pick a group → configure notifications → land on dashboard.

    - **Header pill**: shows current group name; clicking routes to `?mode=switch` picker.
    - **Change group**: pick a different group → views refresh with new scope.
    - **Expired group**: set a group's `end_date` to yesterday in the DB → reload → full-screen redirect to `?mode=expired`.
    - **Admin bootstrap (zero groups)**: log in as admin with no groups in DB → redirected to `?mode=onboarding` → "Create new" button visible → create a group inline → advance through wizard.
    - **Admin management**: navigate to `/app/admin/event-groups` → full table with edit/delete/archive; "Edit" opens `/app/group-settings?groupId=…` without changing the admin's own selected group.

4. **API client regen**: run `just generate-client` after backend changes and check that `frontend/src/client/types.gen.ts` now exposes `selected_event_group_id` on `UserProfile`.
5. **Lint & typecheck**: `just lint` must pass — expect to fix imports of removed views/routes.
