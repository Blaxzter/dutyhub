# Plan: Terminology rename (EventGroup → Event, Event → Task, DutySlot → Shift)

## Context

The current domain vocabulary is ambiguous in both code and UI: `EventGroup` and `Event` sound like near-synonyms, and `DutySlot` is technical jargon. The user wants to adopt clearer, German-translatable terms that match how people already talk about the system:

| Current (EN / DE) | New (EN / DE)                 | Meaning                                                                       |
| ----------------- | ----------------------------- | ----------------------------------------------------------------------------- |
| Event Group / —   | **Event** / **Veranstaltung** | Time-bounded container with a start/end date (e.g. a camp, festival, season)  |
| Event / —         | **Task** / **Aufgabe**        | A specific unit of work inside an Event (e.g. "Set up chairs", "Run kitchen") |
| Duty Slot / —     | **Shift** / **Schicht**       | A bookable time block within a Task (e.g. "Saturday 14:00–16:00, 3 people")   |

This plan is a standalone follow-up to the selection/scoping refactor in [`i-want-to-remove-generic-tome.md`](i-want-to-remove-generic-tome.md) and runs **before** that plan is implemented, so the scoping work uses the new names from day 1.

Blast radius measured from exploration:

- Backend: ~60 Python files, 5 model classes, 42+ Pydantic schemas, 8 CRUD classes, 6 route files, 19 test files.
- Frontend: ~150 Vue/TS files, 200+ auto-generated client types, ~110 i18n keys, 9 route names/URLs.
- DB: 3 table renames + FK column renames + 1 data migration for a `scope_type` enum.
- Hardcoded English strings in `.vue` templates: **0** (everything is i18n — good).

---

## Strategy: three ordered renames in one PR

The renames collide (`EventGroup` → `Event` would overwrite the existing `Event` class). The safe order is:

1. **Rename `Event` → `Task`** (no collision; `Task` doesn't exist yet).
2. **Rename `EventGroup` → `Event`** (safe now that old `Event` is gone).
3. **Rename `DutySlot` → `Shift`** (independent of the above, can go anywhere).

FK column cascades: after step 1, `event_group_id` stays as-is. After step 2, `event_group_id` → `event_id`. After step 3, references to `duty_slot_id` become `shift_id`.

Recommended: **one PR, three commits in that order** — keeps review tractable, preserves bisectability, and leaves the DB in a consistent state at every commit.

## Backend changes

### 1. Database migrations

Create one new Alembic migration (or three sequential — both work; one is tidier) that performs:

```
ALTER TABLE events                RENAME TO tasks;
ALTER TABLE event_groups          RENAME TO events;
ALTER TABLE duty_slots            RENAME TO shifts;

ALTER TABLE tasks   RENAME COLUMN event_group_id   TO event_id;
ALTER TABLE shifts  RENAME COLUMN event_id         TO task_id;
-- bookings, slot_batches, availabilities, event_group_managers, etc. FK columns:
ALTER TABLE bookings          RENAME COLUMN duty_slot_id   TO shift_id;
ALTER TABLE slot_batches      RENAME COLUMN event_id       TO task_id;
ALTER TABLE availabilities    RENAME COLUMN event_group_id TO event_id;
ALTER TABLE event_group_managers RENAME TO event_managers;
ALTER TABLE event_managers    RENAME COLUMN event_group_id TO event_id;
```

Plus indices and constraints: rename PK/FK constraint names so they match the new table names (Postgres doesn't auto-rename them and ugly old names cause confusion in pgAdmin/error messages).

**Data migration for notification scope**: the `notifications.scope_type` column (or wherever `ScopeType = Literal["global", "event_group", "event", "duty_slot"]` lands in the DB) needs a one-time `UPDATE`:

```
UPDATE notifications SET scope_type = 'task'  WHERE scope_type = 'event';       -- step 1
UPDATE notifications SET scope_type = 'event' WHERE scope_type = 'event_group'; -- step 2
UPDATE notifications SET scope_type = 'shift' WHERE scope_type = 'duty_slot';   -- step 3
```

Order matters — do step 1 before step 2 or the first UPDATE overwrites what the second writes.

> ⚠️ **Existing migration files** (6 files under `backend/app/alembic/versions/` that create the old tables) are _not_ edited. They represent historical state and Alembic replays them against empty DBs. Only the new rename migration runs on existing DBs.

## Open questions

1. **`slot_batches` table name**: this table isn't directly in the rename list. Option A — rename to `shift_batches` for consistency; Option B — leave as `slot_batches`. Recommend A.
2. **URL redirect shims**: given gated-community scale, likely no one has bookmarked `/app/event-groups/...`. Recommend skipping redirect shims. Confirm this is acceptable.
3. **`managedEventGroupIds` vs existing `isEventManager` role**: the role name `event_manager` currently refers to the high-privilege admin-ish role. After the rename it becomes ambiguous ("manager of an Event (old EventGroup)" vs "event_manager role"). Options: (a) rename the role to `manager` or `admin_manager`, (b) rename the scoped variable and keep the role name. Recommend (b) with `managedEventIds` + `hasEventManagerRole` to disambiguate.
4. **Notification type code constants**: `EVENT_GROUP_PUBLISHED` → `EVENT_PUBLISHED` collides with the current `EVENT_PUBLISHED`. Keep the rename-through-Task discipline: first rename `EVENT_PUBLISHED` → `TASK_PUBLISHED`, then rename `EVENT_GROUP_PUBLISHED` → `EVENT_PUBLISHED`.
